CTP_PLUS 多节点期货交易系统

简要技术架构说明 · Hub-Spoke 架构 · 多节点水平扩展 · 统一管控

Go (Futures-hub) C++ (Node + Gateway) Etcd · Redis · MySQL

一、项目背景

在期货交易领域，单一 CTP 账户在性能、容量和可靠性方面存在天然瓶颈。当策略规模扩大、品种覆盖增多时，面临以下核心挑战：

⏱️
报单频率受限
CTP 柜台对单账户报撤单频率有严格限制，高频策略容易触发流控
💥
故障单点风险
单节点故障导致全部策略无法交易，缺乏容灾能力
🔗
行情与交易耦合
行情订阅和交易执行共用连接，相互影响性能
📋
管理效率低下
多账户分散管理，缺少统一的监控、路由和风控手段

CTP_PLUS 系统通过 多节点 Hub-Spoke 架构，实现交易能力的水平扩展和统一管控，从根本上解决上述痛点。

二、核心需求

功能需求

需求类别要求
多节点交易支持 N 个交易节点并行运行，每个节点独立连接 CTP 柜台
智能路由根据合约、方向、持仓等规则，自动将订单路由到最优节点
统一风控在中心 Hub 层进行价格偏离、报单频率、净敞口等风控检查
行情独立行情节点与交易节点分离，互不影响
配置管理通过 Web 面板统一管理账户、路由规则、风控参数
实时监控节点状态、持仓、委托、行情的实时查看
幂等防重防止网络重试导致的重复报单

非功能需求

维度要求
高可用节点自动注册与发现，故障自动摘除
低延迟gRPC 通信，跨公网延迟可控
可扩展新增交易节点仅需部署 + 配置，无需修改代码
跨平台Hub 和节点运行在 Linux，策略客户端运行在 Windows

四、核心组件说明

🎯

Futures-hub（中心服务）

Go · go-zero · gRPC :9000 + REST :9200

系统核心中枢。单进程同时提供 gRPC（交易）和 REST（管理）两类接口。负责接收上游策略信号，进行智能路由分发和风控校验，管理下游所有交易/行情节点，并提供 Web 管理面板支持配置、监控、行情查看等功能。

🖧

Futures-node（执行节点）

C++17 · gRPC Server · CTP API

直接对接 CTP 柜台的执行节点。支持双模式运行：Trade 节点 (mode=0) 执行报单、撤单、查仓；MD 节点 (mode=1) 订阅行情并推送 Tick 数据。启动时自动向 Etcd 注册，支持断线重连。

🖥️

FuturesTradeGateway（策略网关）

C++ · Windows · gRPC Client

运行在 Windows 端，对接 MT4/MT5 等策略平台，将交易信号通过 gRPC 跨公网发送到 Hub。支持下单、撤单、状态查询等操作。

📊

CTPMasterMDFeeder（行情采集）

C++ · Windows · MT5 插件

独立的 MT5 行情采集模块，直连 CTP 行情前置获取实时 Tick 数据，转换为 MT5 格式推送给策略端展示。独立运行，不与 Hub 交互。

🌐

Web 管理面板

HTML/JS · 内嵌于 Hub REST 服务

提供节点状态监控、账户启停控制、持仓/委托查看、路由/风控配置和行情订阅管理等功能，实时刷新，统一管理。
