# CTP_PLUS 多节点期货交易系统 — 简要技术架构说明

> **版本**: v1.0 · **日期**: 2026-03-22

---

## 一、项目背景

在期货交易领域，单一 CTP 账户在性能、容量和可靠性方面存在天然瓶颈。当策略规模扩大、品种覆盖增多时，单账户模式面临以下挑战：

- **报单频率受限**：CTP 柜台对单账户报撤单频率有严格限制，高频策略容易触发流控
- **故障单点风险**：单节点故障导致全部策略无法交易，缺乏容灾能力
- **行情与交易耦合**：行情订阅和交易执行共用同一连接，相互影响性能
- **管理效率低下**：多账户分散管理，缺少统一的监控、路由和风控手段

CTP_PLUS 系统正是为解决上述痛点而设计，通过 **多节点、Hub-Spoke 架构**实现交易能力的水平扩展和统一管控。

## 二、核心需求

### 2.1 功能需求

| 需求类别 | 要求 |
|---------|------|
| **多节点交易** | 支持 N 个交易节点并行运行，每个节点独立连接 CTP 柜台 |
| **智能路由** | 根据合约、方向、持仓等规则，自动将订单路由到最优节点 |
| **统一风控** | 在中心 Hub 层进行价格偏离、报单频率、净敞口等风控检查 |
| **行情独立** | 行情节点与交易节点分离，互不影响 |
| **配置管理** | 通过 Web 面板统一管理账户、路由规则、风控参数 |
| **实时监控** | 节点状态、持仓、委托、行情的实时查看与告警 |
| **幂等与防重** | 防止网络重试导致的重复报单 |

### 2.2 非功能需求

- **高可用**：节点自动注册与发现，故障自动摘除
- **低延迟**：gRPC 通信，跨公网延迟可控
- **可扩展**：新增交易节点仅需部署 + 配置，无需修改代码
- **跨平台**：Hub 和节点运行在 Linux 服务器，策略客户端运行在 Windows

## 三、系统架构概览

系统采用 **Hub-Spoke（中心-节点）** 架构，分为四大层次：

```
┌──────────────────────────────────────────────────────────────┐
│  Windows 策略端                                              │
│  ┌─────────────────────┐  ┌──────────────────┐              │
│  │ FuturesTradeGateway │  │ CTPMasterMDFeeder│              │
│  │  (策略信号 → Hub)    │  │ (MT5行情采集,独立) │              │
│  └────────┬────────────┘  └──────────────────┘              │
│           │ gRPC                                             │
├───────────┼──────────────────────────────────────────────────┤
│  中心 Hub │ (Linux 云服务器)                                  │
│  ┌────────▼────────────────────────────────────┐             │
│  │           Futures-hub (Go)                  │             │
│  │  gRPC:9000 (交易) + REST:9200 (管理/行情)    │             │
│  │  路由引擎 · 风控引擎 · 幂等 · 配置管理         │             │
│  └─────┬──────────────┬────────────────────────┘             │
│        │ gRPC         │ gRPC                                 │
│  ┌─────▼──────┐ ┌─────▼──────┐                              │
│  │Trade Node×N│ │  MD Node   │    ← Futures-node (C++)      │
│  │ 报单/撤单   │ │ 行情订阅    │                               │
│  └─────┬──────┘ └─────┬──────┘                              │
│        │ CTP API      │ CTP API                              │
├────────┼──────────────┼──────────────────────────────────────┤
│  ┌─────▼──────────────▼──────┐                              │
│  │     CTP 交易所 (SimNow / 实盘)                             │
│  └───────────────────────────┘                              │
├──────────────────────────────────────────────────────────────┤
│  基础设施: Etcd (服务发现) · Redis (缓存/幂等) · MySQL (持久化) │
└──────────────────────────────────────────────────────────────┘
```

## 四、核心组件说明

### 4.1 Futures-hub（中心服务）

- **技术栈**：Go / go-zero 框架
- **职责**：系统的核心中枢，单进程同时提供 gRPC 和 REST 两类接口
- **核心能力**：
  - 接收上游策略信号，进行路由分发和风控校验
  - 管理下游所有交易/行情节点
  - 提供 Web 管理面板，支持配置、监控、行情查看

### 4.2 Futures-node（执行节点）

- **技术栈**：C++17 / gRPC Server
- **职责**：直接对接 CTP 柜台，执行具体的交易和行情操作
- **双模式**：
  - **Trade 节点 (mode=0)**：执行报单、撤单、查仓、查资金
  - **MD 节点 (mode=1)**：订阅行情并推送 Tick 数据到 Hub
- 启动时自动向 Etcd 注册，支持 CTP 断线重连

### 4.3 FuturesTradeGateway（策略网关）

- **技术栈**：C++ / Windows
- **职责**：运行在 Windows 端，对接 MT4/MT5 等策略平台，将交易信号通过 gRPC 发送到 Hub

### 4.4 CTPMasterMDFeeder（行情采集）

- **技术栈**：C++ / Windows / MT5 插件
- **职责**：独立的 MT5 行情采集模块，直连 CTP 行情前置，**不与 Hub 交互**

### 4.5 Web 管理面板

- **技术栈**：HTML/JS，内嵌于 Hub 的 REST 服务
- **职责**：节点状态监控、账户启停、持仓/委托查看、路由/风控配置、行情订阅管理

## 五、核心数据流

### 5.1 订单流

```
策略端 → Hub(幂等+风控+路由) → 目标交易节点 → CTP交易所
                                    ↓ 回报
CTP交易所 → 交易节点 → Hub(持仓更新+落库) → 策略端
```

### 5.2 行情流

```
CTP行情前置 → MD节点 → Hub → Redis缓存 → Web面板展示
```

### 5.3 节点管理流

```
节点启动 → Etcd注册 → Hub Watch感知 → 写入MySQL → Web面板展示状态
节点下线 → Etcd Lease过期 → 自动标记offline
```

## 六、部署架构

| 部署位置 | 组件 | 说明 |
|---------|------|------|
| **Linux 云服务器** | Futures-hub | gRPC :9000 + REST :9200 |
| | Futures-node ×N | Trade节点 :57001~N，MD节点独立 |
| | Etcd / Redis / MySQL | 基础设施 |
| **Windows 本地** | FuturesTradeGateway | 策略信号源，通过公网 gRPC 连接 Hub |
| | CTPMasterMDFeeder | MT5 行情采集，独立运行 |
